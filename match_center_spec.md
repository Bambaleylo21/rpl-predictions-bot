# Техническая спецификация «Матч-центр»

ТЗ на основе реально работающей реализации в текущем боте — для повторения аналогичного раздела в другом проекте.

## 1. Назначение и точка входа

Матч-центр — отдельный полноэкранный раздел, открывающийся по тапу на карточку конкретного матча (из разделов «Матчи» или «Таблица»). В Telegram Mini App подключена нативная кнопка «Назад» (BackButton) — закрытие экрана ощущается как обычная навигация, а не веб-попап. Экран показывает один конкретный матч и подгружает данные только при открытии — никакого фонового опроса, пока пользователь не зашёл именно в этот матч.

## 2. Backend: два endpoint'а, вызываются параллельно при открытии

**`GET /api/miniapp/match/center?t=<турнир>&match_id=<id>`** — основной, агрегирует всё в один ответ.

**`GET /api/miniapp/match/predictions?match_id=<id>`** (условное название) — отдельно, прогнозы всех участников турнира на этот матч («Прогнозы соперников»): до начала матча видно только количество и % по исходам, после начала — сами прогнозы, после завершения — у кого сколько очков и как изменилось место в таблице.

### Состав ответа основного endpoint'а

- `home_team`, `away_team`, `kickoff`, `home_score`, `away_score` — базовые данные матча из своей БД.
- `standings.home` / `standings.away` — место, очки, сыграно (текущая позиция каждой команды в официальной таблице лиги).
- `standings_table` — полная официальная таблица лиги (для вкладки «Таблица» внутри матч-центра).
- `h2h` — последние 5 личных встреч этой пары команд (дата, счёт).
- `form.home` / `form.away` — последние 5 матчей команды в любых турнирах (для кружков-индикаторов под гербами в шапке).
- `lineups` — стартовые составы, если уже опубликованы, с обогащением: у каждого игрока голы/передачи/рейтинг за сезон.
- `ai_estimate` — процентная оценка исходов (см. п. 3, собственная модель).
- `accuracy.home` / `accuracy.away` — личный % угадывания текущего пользователя по каждой из команд (среди уже рассчитанных его прогнозов на матчи этой команды).
- `injuries` — травмы/дисквалификации на матч.
- `events` — хронология (голы/карточки/замены с минутой).
- `statistics` — владение, удары, угловые, фолы, карточки (если матч идёт/прошёл).

### Кэширование и живой режим

Все внешние вызовы к футбольному API идут через общий слой кэша (детали — в отдельном описании инфраструктуры запросов), с разными TTL по типу данных: таблица лиги — 3 часа, состав команды/id команд — 24 часа, H2H — 6 часов, форма — 2 часа, статистика игроков — 12 часов, травмы — 6 часов, составы — 10 минут.

Отдельно — «живое окно»: матч считается live, если кикофф уже прошёл, но ещё не вышел за разумное окно длительности (кикофф + 130 минут) и финальный счёт ещё не проставлен синхронизацией. Для live-матчей события и статистика обновляются раз в 90 секунд вместо обычных 10 минут — чтобы картина была почти в реальном времени, но без лишних затрат вне окна игры.

## 3. Собственная модель «Оценка ИИ» (без внешних коэффициентов)

Вместо букмекерских коэффициентов/чужого `/predictions` (оказались недоступны или ненадёжны для нашей лиги) — своя упрощённая Пуассон-модель силы атаки/обороны, не требующая отдельного внешнего запроса вообще:

1. **Источники данных**: исторические результаты прошлых сезонов (разовый бэкфилл через API) + результаты уже сыгранных матчей текущего сезона (из собственной БД). Текущий сезон весит в 2.5 раза больше исторических данных (свежая форма важнее старой) — веса настраиваемые константы.
2. **Attack/defense рейтинг команды**: отношение среднего числа забитых (attack) / пропущенных (defense) голов команды к среднему по лиге, с усадкой (shrinkage) к нейтральному значению 1.0 — команды с малой историей (новички, мало сыгранных матчей) не получают экстремальных оценок на 1-2 матчах. Формула усадки: `(raw * played_weight + 1.0 * k) / (played_weight + k)`, где k — константа регуляризации (в нашем случае 8 «виртуальных» матчей).
3. **Ожидаемые голы (λ) на конкретную пару**: `λ_дома = среднее_голов_дома_по_лиге × attack(хозяев) × defense(гостей)`, аналогично для гостей.
4. **Вероятности исходов**: по распределению Пуассона от λ_дома и λ_гостей считается матрица вероятностей всех разумных счётов, из неё — вероятность победы хозяев / ничьей / победы гостей (суммирование по диагоналям матрицы).
5. Команды, для которых вообще нет данных, получают нейтральную силу (attack=defense=1.0).

Важно: это справочная статистическая оценка, не выдаётся и не подписывается как «прогноз/коэффициент» — на фронтенде явно подписано как приблизительная оценка на основе статистики.

## 4. Frontend: структура экрана

**Шапка**: гербы команд, счёт (или кикофф, если матч не начался), позиция в таблице рядом с названием команды (если есть), кружки формы (5 последних результатов, самый свежий — крайний справа, цвета по W/D/L).

**Вкладки** (нижний таб-бар внутри экрана):
1. **Детали** — карточка «Мой прогноз», хронология событий матча, карточка «Процент угадывания» (личная статистика по каждой команде), карточка «Прогнозы соперников» (см. п. 2, второй endpoint).
2. **Встречи** — последние личные встречи (H2H).
3. **Таблица** — полная официальная таблица лиги.
4. **Составы** — стартовые составы с номером, позицией и статистикой игрока за сезон (голы/передачи/рейтинг), если уже опубликованы API.
5. **Статистика** — владение мячом, удары, угловые, фолы, карточки (только для матчей, которые идут или завершились).
6. **Оценка ИИ** — три процента (домашняя победа / ничья / победа гостей) из собственной модели (п. 3).

Каждая вкладка рендерится условно (`matchCenterTab === '...'`), данные для всех вкладок приходят одним запросом при открытии экрана (кроме «Прогнозов соперников» — отдельный параллельный запрос).

## 5. Что нужно для повторения в другом боте

- Тот же принцип: один агрегирующий backend-endpoint на весь экран + один отдельный на список прогнозов участников, оба вызываются один раз при открытии экрана, без поллинга.
- Слой кэша с TTL по типам данных (см. п. 2) поверх обёртки вызовов к футбольному API.
- Если нужна колонка "Оценка ИИ" без реальных коэффициентов — реализовать Пуассон-модель по п. 3 (не требует доп. запросов к внешнему API, полностью считается на своих же исторических+текущих результатах).
- Frontend — вкладочная структура внутри одного экрана, а не отдельные экраны на каждую вкладку (упрощает навигацию и переиспользование шапки/данных).
